# Запрос

## Общая информация

*запросах*в спецификации интерфейсаВ рамках взаимодействия с платежной платформой через statistics_api все данные от сервиса мерчанта должны передаваться в— сообщениях HTTP заданной структуры — с использованием метода POST. Описание общей структуры таких запросов представлено далее, а описание структур данных для конкретных запросов —.

## Структура

В каждом запросе к платежной платформе должны передаваться следующие элементы в указанном порядке:

`Host`[в HTTP версии 1.1](https://tools.ietf.org/html/rfc2616#page-31)Далее представлен пример запроса с рекомендуемым набором полей заголовка и обязательными параметрами. Содержимое JSON-строки в этом примере разбито на несколько строк для удобства чтения.В дополнение к обязательному полюв заголовке можно использовать любые другие поля из числа допустимых.

- `POSTv1/operations/getHTTP/1.1`стартовая строка с указанием метода передачи запроса () и конечной точки в интерфейсе statistics_api (например,), протокола и его версии ();
- `Host``basic_data_api_text_link`()заголовок с полем, содержащим доменное имя для запросов через statistics_api;
- пустая строка — разделитель, отделяющая служебную информацию от тела сообщения;
- тело сообщения, содержащее JSON-строку в кодировке UTF-8 с набором данных и подписью к ним.

## Параметры адресации

`basic_data_api_text_link`При формировании запросов необходимо указывать базовый и относительный адреса отправки. В качестве базового адреса для запросов через statistics_api используется доменное имя, а в качестве относительного — указатель конечной точки в интерфейсе в соответствии со спецификацией.

::: tip
`https://{доменное имя платформы для запросов через statistics_api}/{указатель конечной точки}``https://basic_data_api_text_link/v1/balance/get`. Например, адрес для запроса на получение балансов по проектам мерчанта выглядит какПолный адрес в этом случае представляет собой строку вида, но в таком виде при работе с POST-запросами полные адреса не используются.
:::

## Тело

`"<название параметра>": <значение параметра>`ru_db_api_token.ditaru_db_signature_intro.ditaВ теле сообщения должна содержаться JSON-строка с набором данных в формате. Чтобы предотвратить утечку и подмену данных во время их передачи в платежную платформу, в составе JSON-строки используются токен и подпись, а на транспортном уровне передачи — протокол TLS 1.2, обеспечивающий шифрование. Подробная информация о формировании токена представлена в пункте, а о формировании подписи — в разделе.
