# Quickstart

This quickstart is for teams that want to launch Gate with the smallest possible amount of reading and the highest possible amount of reusable code.

## What you get from this guide

After completing this guide, you will have:

- a signed server-to-server payment request
- a synchronous API response from Gate
- a callback endpoint that receives the final payment result

## Before you start

Prepare the following:

- `project_id` for your test project
- `secret_key` for signing requests and validating callbacks
- the Gate base URL for your environment
- a callback URL available to the platform
- a test checkout page or API endpoint in your web service

## Where `project_id` and `secret_key` come from

For the first integration, these values are not generated by your code.

- `project_id` is the identifier of the merchant project assigned by Platform during onboarding.
- `secret_key` is the signing key issued for the same project and environment.

If you do not have them yet, the next action is simple: ask your Platform account manager or technical support to provision a test project and share the test `project_id` and `secret_key` for API integration.

## If you only need the first request

Go straight to [**Basic implementation**](./gate-quickstart-basic-implementation.md). That page shows:

- the exact Gate URL
- the exact JSON payload shape
- how to generate the signature
- how to accept the callback

## Best path for first integration

1. Read [**Integration overview**](./gate-how-to-integrate.md) if you need the architecture in one page.
2. Go straight to [**Basic implementation**](./gate-quickstart-basic-implementation.md) for copy-paste code.
3. After the first successful request, continue with [**One-time purchases**](./gate-purchase.md), [**Request format**](./gate-requests-format.md), and [**Signature generation**](./signature-gate.md).

## Recommended implementation style

For a first launch, use the shortest possible path:

- build and sign the final JSON payload on the server side
- send a one-step purchase request to Gate
- treat callbacks as the source of truth for payment status

Once that works, add two-step flows, refunds, payouts, 3-D Secure handling, and any other payment-specific procedures.
