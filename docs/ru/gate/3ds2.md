# Формат уведомления о результате аутентификации

## Формат уведомления для аутентификации по базовой схеме

`pares`При использовании базовой схемы аутентификации через Gate данные о результате аутентификации передаются в веб-сервис в формате платежной платформы. Как и при аутентификации 3ds1, данные о результате аутентификации передаются в параметре, однако для 3ds2 используется другой формат этих данных.

## Формат уведомления для аутентификации по расширенной схеме

`cres`При использовании расширенной схемы аутентификации через Gate данные о результате аутентификации передаются в веб-сервис в формате эмитента. Эти данные передаются в параметреуведомления.

## Формат уведомления о приеме данных об устройстве пользователя

`3ds_notification_url`Если в запросе на проведение платежа передан параметр, то при расширенной схеме аутентификации через Gate сервер управления доступом (ACS) эмитента передает в веб-сервис уведомление о получении данных об устройстве пользователя (в формате эмитента), в противном случае уведомление передается в платежную платформу.

## Формат оповещения о результате платежа

`mpi_result`В случае выполнения аутентификации 3ds2 при проведении платежа в объектеоповещения дополнительно могут передаваться следующие параметры:

По умолчанию эти параметры не передаются. Для того, чтобы получать их в оповещениях, необходимо обратиться в службу технической поддержки support@example.com

- `mpi_operation_id`— идентификатор операции на стороне 3DS‑сервера;
- `ds_operation_id`— идентификатор операции на стороне cервера каталогов международной платежной системы;
- `acs_operation_id`— идентификатор операции на стороне сервера управления доступом эмитента;
- `mpi_timestamp`— дата и время аутентификации;
- `cardholder_info`— информация об аутентификации, которую рекомендуется отобразить пользователю при уведомлении о результате проведения платежа;
- `authentication_flow0102`— указатель использованного варианта аутентификации:для frictionless flow илидля challenge flow.

## Формат запроса для перенаправления пользователя

[Информация о запросе уточняется]При работе через Gate для перенаправления пользователя на страницу ожидания необходимо отправить HTTP-запрос с использованием информации из оповещения с данными для перенаправления.

## Оповещения с данными для перенаправления

### Оповещение для аутентификации по базовой схеме

ru_gate_callbacks.ditaПри использовании базовой схемы аутентификации через Gate платежная платформа передает в веб-сервис оповещение с данными для перенаправления пользователя. Оповещение передается в стандартном формате, описание которого представлено в разделе.

`424242******4243customer_12acs`В следующем примере данные из оповещения свидетельствуют о том, что для платежной карты с номеромпользователяподдерживается аутентификация 3ds2 и необходимо перенаправить пользователя на страницу ожидания платежной платформы. Оповещение содержит объектс необходимыми для этого данными: сообщением PAReq, адресом страницы ожидания платежной платформы и данными мерчанта в платежной системе.

### Оповещения для аутентификации по расширенной схеме

ru_gate_callbacks.ditaПри использовании расширенной схемы аутентификации через Gate от платежной платформы к веб-сервису могут передаваться оповещения с данными для перенаправления пользователя на страницу аутентификации (ACS URL) эмитента. Оповещения передаются в стандартном формате, описание которого представлено в разделе. Оповещения с данными для перенаправления пользователя на страницу аутентификации не передаются в случаях, если на стороне эмитента выбран вариант аутентификации frictionless flow. Вместо такого оповещения со стороны веб-сервиса необходимо принять и обработать оповещение о результате проведения платежа.

`urlactioninput`Следующий пример содержит данные для перенаправления пользователя на страницу аутентификации. Эти данные необходимо использовать следующим образом: адрес из параметра— в атрибуте, а данные из других параметров — в тегах.
